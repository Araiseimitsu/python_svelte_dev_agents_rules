# AGENTS.md — AI Agent Rules

本ファイルは、本プロジェクトにおける AI Agent の最優先ルール（正本）である。
Claude Code は `CLAUDE.md` の `@AGENTS.md` インポート経由で本ファイルを読み込む。
すべてのエージェント（Claude / Codex / OpenCode / Antigravity など）は本ファイルの同一ルールに従う。

**本ファイルの記載方針（メンテナンス時も厳守）**:
エージェントが自力で導けない「プロジェクト固有の決定事項」と「安全ゲート」のみを記載する。
一般的なベストプラクティス（良い命名・例外処理の徹底など、現行のエージェントが既定で行うこと）は書かない。
重複記載は矛盾の温床になるため、各ルールの正本は1ファイルに固定し、他所からは参照する。
機械検査できるルール（依存方向・行数制限・fetch 直呼び禁止など）は、実装開始後に lint / CI（例: `import-linter`、`eslint-plugin-boundaries`、`dependency-cruiser`）へ移管し、**移管が完了したルールは本文書から削除する**。文章は最弱の強制手段であり、機械が止める仕組みを常に優先する。

- [ ] 破壊的操作・事前確認対象を含む場合、ユーザー承認を得る段取りにしたか（§14 安全確認）
- [ ] フロントエンド作業なら `.docs/frontend-svelte.md` を、Backend の実装・構成変更なら `.docs/backend-python.md` を読んだか
- [ ] 完了報告の前に: 検証（テスト・lint・型チェック）の実行結果があるか（§3 開発プロセス）、学びがあれば `.docs/memory/` に記録したか

## 1. ルール優先順位

1. ユーザーの明示指示
2. 本ファイル `AGENTS.md`
3. `.docs/workflow.md`（開発プロセスの正本）および `.docs/frontend-svelte.md`（フロントエンド作業時のみ）
4. 既存コードの設計・命名・構造
5. 各エージェントの Global Rule

ただし、破壊的操作・機密情報・DB操作・外部送信については、上記に関わらず常に安全側を優先する。

## 2. 基本方針

- すべての回答・コメント・ドキュメントは **日本語** で書く。
- 依存は一方向、責務は小さく明確に。「動けばよい」ではなく、後から直せる構造にする。

## 3. 開発プロセス

開発プロセス（理解→計画→実装→検証→報告の手順、完了の定義、デバッグ規律、ループ開発手順）の正本は `.docs/workflow.md` である。以下は絶対規律のみ。

- **検証なしに「完了」と報告しない。** テスト・lint・型チェックの実行結果が完了の証拠である。
- 業務ロジック・判定処理・データ変換は **テストファースト** で実装する。
- **同一アプローチで3回失敗したら停止**し、再調査するかユーザーに報告する。
- **基本はシングルエージェントで進める。** マルチエージェント活用・ループ開発はユーザーの明示的な依頼時のみ（§16 マルチエージェント）。
- 影響範囲が想定より大きい場合・仕様の解釈が分かれる場合は、進めずに確認する。

## 4. プロジェクト構成

### レイアウト

```txt
project-root/
├─ backend/
├─ frontend/
├─ docker/
├─ tests/
├─ .docs/
├─ README.md
├─ AGENTS.md
└─ CLAUDE.md（@AGENTS.md を参照するのみ）
```

この構造を勝手に崩してはいけない。

### ディレクトリ責務

| パス | 役割 |
| --- | --- |
| `backend/` | FastAPI / Python 側の実装 |
| `frontend/` | Svelte / Vite 側の実装 |
| `docker/` | Docker 関連ファイル |
| `tests/` | 統合テスト、E2Eテスト、横断的なテスト |
| `.docs/` | 設計ルール、変更履歴、開発ドキュメント |
| `README.md` | セットアップ、起動方法、環境変数の説明 |
| `AGENTS.md` | 本プロジェクトの AI Agent ルール（正本） |

### 実装コードの配置

実装コードは必ず各プロジェクト配下の `src/` に置く（`backend/src/`・`frontend/src/`、テストは各 `tests/`）。
プロジェクトルートへの実装コード直置きは禁止する。例外は、設定ファイル・README / ドキュメント・Docker 関連ファイル・起動スクリプト・AGENTS.md / CLAUDE.md のみ。

## 5. Backend ルール

Python バックエンドは FastAPI を基本とする。

- **必須**: Python 管理は `uv`。`pyproject.toml` + `uv.lock`（`uv.lock` は必ずコミット）。依存追加は `uv add`、実行は `uv run`、テストは `uv run pytest`。
- **禁止**: `pip` 単独使用、`poetry` 併用、`requirements.txt` の手動管理、直接 `python` 実行、グローバル Python 環境への依存。
- コマンド例・推奨構造（api / application / domain / infrastructure / core / shared の層構成）は `.docs/backend-python.md` を参照する。Backend の実装・構成変更時は必ず参照すること。

---

## 6. Frontend ルール

- フロントエンドは `frontend/` 配下で管理する。
- Node.js パッケージ管理は `pnpm` を使用し、`pnpm-lock.yaml` はコミット対象とする。
- Svelte / SvelteKit / Vite 固有のルールは `.docs/frontend-svelte.md` に従う。フロントエンド作業時は必ず参照する。
- 拡張ルール（`.docs/frontend-*.md`）は本ファイルより優先される。拡張ルールが無いのに UI フレームワーク前提の構造を作ってはいけない。

---

## 7. 依存方向

依存関係は一方向にする。

- **Backend**: `api → application → domain`、`application → infrastructure`。`shared` は全層から参照可能。
- **Frontend**: `Presentation → State → Application → Infrastructure`。`Shared` は全層から参照可能。
- **禁止**: レイヤー逆流、循環依存、UI から Infrastructure 直呼び、API 層に業務ロジック、Infrastructure 層に画面都合の処理。
- **空のパススルー層を作らない。** 小規模プロジェクトでは隣接層（例: Backend の application と domain）を統合してよい。ただし依存の一方向性は維持する。層構造の遵守より、中身のある層だけを持つことを優先する。

---

## 8. 二重実装・責務重複の防止

- 同じ責務を複数レイヤー・複数技術で重複実装しない。既存実装がある場合は、追加実装の前に責務の所在を確認する。
- 特に **API呼び出し・入力検証・状態管理・データ変換・認証・ルーティング** は、正本となる実装箇所を1つに固定する。
- FastAPI をバックエンドとする構成では、SvelteKit のサーバー機能を安易に追加して API や業務ロジックを重複実装しない。
- やむを得ず類似処理を追加する場合は、既存実装を共通化・移設できない理由をコメントまたはドキュメントで明示する。

---

## 9. 環境変数

環境変数は用途ごとに分離する: `backend/.env`（FastAPI 用）、`frontend/.env`（Svelte / Vite 用）、root `.env`（Docker Compose や全体起動用）。

- `.env` は Git 管理しない。`.env.example` は Git 管理する。
- DB接続情報、APIキー、秘密鍵、NAS認証情報は `backend/.env` に置く。frontend には置かない。
- frontend に置く値は、ブラウザから見えてよいものだけ。接頭辞は `VITE_`。
- 記入例と `.gitignore` 必須例は `.docs/env-examples.md` を参照する。`.env` / `.env.example` の作成・変更時は必ず参照すること。

## 10. Docker ルール

Docker 内でも依存管理は各技術の標準に従う。Backend は `uv`（`uv.lock` ベースで同期）、Frontend は `pnpm-lock.yaml` ベースで同期する。

## 11. コード品質・ログ

- 1ファイル 1,000行以内（`.svelte` は 300行以内: `.docs/frontend-svelte.md`）。
- 共通処理は別モジュールへ分離し、重複・冗長な構造を作らない。
- **秘密情報をログに出してはいけない。**
- エラーの握りつぶし（catch して無視・テストの skip・lint ルールの安易な無効化）で検証を通してはいけない。

## 12. テスト

テスト導入を前提とし、以下の基準で対象を判断する。

| 区分 | 対象 |
| --- | --- |
| 必須 | 業務ロジック、判定処理、API 入出力、Application 層、データ変換処理、ファイル操作の重要処理 |
| 不要 | UI表示のみ、試作コード、一度きりの処理、スタイルのみの変更 |

```bash
uv run pytest   # backend
pnpm test       # frontend
```

## 13. ドキュメント

重要な変更履歴は `.docs/update.md` のみに記録する。

| 区分 | 内容 |
| --- | --- |
| 記録する | 仕様変更、重要な設計変更、ディレクトリ構成変更、DB構造変更、API仕様変更、運用上の注意点 |
| 記録しない | 試行錯誤、一時ログ、単なるエラー履歴、作業メモ、その場限りの検証内容 |

README.md には必ず、セットアップ手順・実行方法・テスト方法・必要な環境変数・ディレクトリ構成・開発時の注意点を書く。

## 14. 安全確認（破壊的操作）

DB の削除・初期化・リセットなど破壊的操作は、**影響範囲を明示し、ユーザー承認を得てから** 実行する。承認なしでは実行しない。

本プロジェクトでは特に以下を事前確認対象とする。

- `.env` 変更
- lock ファイル更新
- DB構造変更
- ディレクトリ構成変更
- 大量ファイル生成
- Docker 設定変更

---

## 15. コード簡素化（code-simplifier）

以下の **いずれかに該当する場合、タスク完了前に自動で実行する**。該当しない小さな変更では実行不要であり、儀式的な「確認しました」報告をしてはいけない。

- 100行以上の変更があった場合
- 新規モジュールを追加した場合
- 可読性の低下、責務の混在、重複の増加、既存構造との不一致が見られる場合

実行ルール:

- **実装したエージェント自身がインラインで実施する**（サブエージェントへ委譲しない）。観点を「実装者」から「簡素化レビュー」に切り替えて変更箇所を読み直す。
- 仕様を変更しない（API・DB・入出力形式はすべて維持）。既存テストの通過を確認する。
- ユーザー承認なしの大規模リファクタは禁止。

## 16. マルチエージェント・ループ開発（完全オプトイン）

- **既定は常にシングルエージェント**（理解→計画→実装→検証→報告）。文脈を1つのエージェントが保持し続ける方が、精度・速度ともに優れる。
- マルチエージェント活用・ループ開発（実装→レビュー→修正の自律ループ）は、**ユーザーが明示的に依頼した場合のみ** 実行する。エージェント側から提案・示唆してはいけない。一度の依頼は当該タスク限り。
- 例外として、広範囲のコードベース調査の探索委譲と、時間のかかる処理のバックグラウンド実行は依頼なしで活用してよい。
- 各エージェントのグローバルルールに自動委譲トリガー（例: 失敗回数による外部 AI への委譲）がある場合も、本節が優先される。ユーザーの明示的な依頼（「ask GPT」「codex に聞いて」等）があった場合のみ委譲する。
- ループ中も §14 安全確認の事前確認・破壊的操作を自動実行しない。レビュー指摘の握りつぶし・検証の省略で「収束した」と報告しない。
- ループ手順・終了条件の正本は `.docs/workflow.md` §6 である。

## 17. メモリ・自己成長

各エージェントは開発中の学びを **リポジトリ内の共有ファイル `.docs/memory/`** に記録し、次のタスクに活かす。全エージェントが同じメモリを読み書きする。

```txt
.docs/memory/
├─ MEMORY.md          # インデックス（1行1メモ）。タスク開始時に必ず読む
├─ pitfalls.md        # 失敗パターンと回避策
├─ environment.md     # 環境固有の知識（OS・ツール・パス等）
├─ decisions.md       # 設計判断と理由
└─ preferences.md     # ユーザーの好み・指摘履歴
```

- **参照**: タスク着手時（理解フェーズ）に `MEMORY.md` を読み、関連カテゴリファイルを参照してから作業を始める。
- **記録**: ①ユーザーからの指摘・修正 → `preferences.md`、②失敗原因の判明 → `pitfalls.md`、③環境固有のハマりどころ → `environment.md`、④重要な設計判断 → `decisions.md`。記録したら `MEMORY.md` に1行のインデックスを追記する。
- **記録禁止**: 秘密情報・個人情報、コードや既存ドキュメントから読み取れる事実、一時的な作業メモ。
- **淘汰**: 矛盾する古いメモは更新または削除する。同種のメモは統合し、追記だけで肥大化させない。**`MEMORY.md` のインデックスは最大20行**とし、超える場合は追記の前に統合・削除を行う（機械的上限であり例外なし）。
- `.docs/update.md` との棲み分け: `update.md` は仕様・構成の変更履歴（What）、`memory/` はエージェントの行動改善知識（How）。

## 18. Google Workspace CLI（`gws`）

- **書き込み系（send/create/update/delete/insert/patch/batchUpdate）は必ず事前確認**。実行前に対象アカウント・対象データ・影響範囲を明示すること。
- コマンド形式・実行例・自動許可スコープの詳細は `.docs/providers.md` を参照する。`gws` 使用時は必ず参照すること。

## 20. 禁止事項まとめ

- 検証（テスト・lint・型チェック）なしに完了報告すること
- 拡張ルールが無いのに UI フレームワーク前提の構造を作ること
- ユーザー承認なしで DB の破壊的操作を行うこと
- 仕様変更を伴う自動リファクタリングを行うこと
- 同じ責務の処理を別レイヤー・別技術に重複実装すること
- FastAPI が存在する構成で、同等の API や業務ロジックを SvelteKit 側に重ねて実装すること
- プロジェクトルートへの実装コード直置き（§4 の例外を除く）
- 秘密情報の frontend 配置・ログ出力・外部送信
- エラーの握りつぶし・テストの skip・lint ルールの安易な無効化で検証を通すこと
- ユーザーの明示的な依頼なしでループ開発（自律レビューループ）・マルチエージェント並列実装を開始・提案すること
- `.docs/memory/` に秘密情報・個人情報を記録すること
