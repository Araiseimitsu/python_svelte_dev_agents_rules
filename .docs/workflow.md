# Development Workflow — AI 開発プロセスルール

このファイルは、AI Agent が開発を進める際のプロセス（動き方）を定義する。
構造・品質・安全のルールは `AGENTS.md` に従う。

---

## 1. 基本ループ

すべてのタスクは以下のループで進める。

```txt
理解 → 計画 → 実装 → 検証 → 報告
```

| フェーズ | やること |
| --- | --- |
| 理解 | 既存コード・既存実装・関連ドキュメントを確認し、影響範囲を見積もる。`.docs/memory/MEMORY.md` を読み、関連メモを参照する |
| 計画 | 複数ファイルに跨る変更・仕様に関わる変更は、実装前に計画を提示する |
| 実装 | 小さい単位で実装し、単位ごとに動作を確認する |
| 検証 | テスト・lint・型チェックを実行し、結果を確認する |
| 報告 | 実行した検証と結果（証拠）を添えて完了を報告する。学びがあれば `.docs/memory/` に記録する（`AGENTS.md` §19） |

### 着手前の必須確認

- `.docs/memory/MEMORY.md` を確認し、関連する過去の学び（失敗パターン・環境知識・ユーザーの好み）を把握する。
- 同じ責務の既存実装が無いか確認する（`AGENTS.md` §9 二重実装防止）。
- 変更対象がどのレイヤーの責務かを判断してから書き始める。
- 要求が曖昧な場合は、仮定で進めず先に質問する。

---

## 2. 完了の定義（Definition of Done）

以下をすべて満たすまで「完了」と報告してはいけない。

1. テストが通過している（`uv run pytest` / `pnpm test`）。
2. lint・型チェックがエラーなしである（プロジェクトに設定されたもの。例: `ruff` / `mypy` / `svelte-check` / `eslint`）。
3. 変更が要求を満たすことを、実際の実行結果で確認した。
4. 「たぶん動く」「動くはず」での完了報告は禁止。**検証コマンドの実行結果が証拠**である。

テスト・lint が未整備のプロジェクトでは、最低限の手動動作確認を行い、その内容を報告に含める。

---

## 3. テストファースト

- 業務ロジック・判定処理・データ変換は、**先に失敗するテストを書き、通す形で実装する**。
- バグ修正は、まずバグを再現するテストを書き、修正後にそのテストが通ることを確認する。
- テスト不要な対象（UI表示のみ・試作コード等）は `AGENTS.md` §13 の基準に従う。

---

## 4. デバッグ規律

- エラーが出たら、**修正を試す前に原因を特定する**。ログ・スタックトレース・再現条件を先に確認する。
- 推測で修正を積み重ねない。1つの仮説につき1つの変更で検証する。
- **同一アプローチで3回失敗したら停止**し、以下を行う。
    1. 仮説を白紙に戻して再調査する、または
    2. 試したこと・分かったこと・詰まっている点を整理してユーザーに報告する。
- 失敗したまま別の場所を書き換えて「動いたことにする」のは禁止。
- エラーの握りつぶし（catch して無視・テストのskip・lintルールの安易な無効化）で検証を通すことは禁止。

---

## 5. タスク分割・並列化・委譲

**基本はシングルエージェントで進める。** 文脈を1つのエージェントが保持し続ける方が精度・速度ともに優れる。委譲は以下の限定的な場面のみとする。

### 委譲してよい場面（依頼なしで可）

- **広範囲のコードベース調査**（多数ファイルの横断検索・構造把握）→ 探索用エージェントに委譲し、結論だけ受け取る。
- **時間のかかる処理**（ビルド・全体テスト等）→ バックグラウンド実行し、待ち時間に他の作業を進める。

### 委譲しない場面

- 実装タスクそのもの（数ファイルの修正から複数レイヤーの変更まで）。文脈の分断による精度低下と往復オーバーヘッドの方が大きい。
- 順序依存のある一連の変更（分割すると整合性が壊れる）。
- 破壊的操作・承認が必要な操作を含むタスク（`AGENTS.md` §16 に従い、自動委譲しない）。
- 実装の並列分担・レビュー委譲は、ユーザーが明示的に依頼した場合のみ行う（§6）。

### 分割の原則

- タスクは「単独で検証可能な単位」で分割する。
- 分割した各タスクにも本ワークフロー（検証・完了の定義）を適用する。

---

## 6. ループ開発（実装→レビュー→修正の自律ループ）— 完全オプトイン

**ユーザーが明示的に依頼した場合のみ** 実行する（`AGENTS.md` §18）。エージェント側から提案・示唆してはいけない。依頼が無ければ、通常フロー（理解→計画→実装→検証→報告）で進める。

### ループ手順

```txt
実装 ＋ 検証パス（テスト・lint・型チェック）
↓
レビュー（既定: 観点を切り替えた自己レビュー。委譲はユーザー指定時のみ）
↓
Critical 指摘の修正 ＋ 検証パス
↓
再レビュー（指摘が残る場合のみ繰り返し）
```

- レビュー担当は **自己レビューを既定** とする（レビュー観点を明示的に切り替え、実装時の判断を前提にせず読み直す）。文脈の再説明コストと往復レイテンシを避けるためである。
- 別プロバイダー（例: `mcp__codex__codex` 経由の Codex）やサブエージェントへのレビュー委譲は、ユーザーが依頼時に指定した場合のみ行う。
- 指摘は Critical（修正必須: バグ・セキュリティ・仕様逸脱）と Recommendation（推奨: 改善提案）に区分する。
- 各周回とも、修正後に §2 完了の定義の検証を通してから次のレビューに進む。

### 終了条件

1. **正常終了**: Critical 指摘がゼロになった時点で終了する。Recommendation は完了報告に添えてユーザーの判断に委ねる。
2. **上限到達**: **最大3周**で収束しない場合は停止し、残っている指摘・試したこと・詰まっている点を整理してユーザーに報告する（§4 の「3回失敗で停止」と同じ規律）。

### メモリとの接続

- ループ中に判明した失敗原因・環境固有の問題・設計判断は、`AGENTS.md` §19 に従い `.docs/memory/` に記録する。
- 同じ指摘を別タスクで繰り返し受けた場合は、必ず `pitfalls.md` または `preferences.md` に記録し、再発を防ぐ。

---

## 7. Git 運用

- 意味のある作業単位で小さくコミットする。1コミット = 1つの論理的変更。
- コミット前に §2 の検証を通す。壊れた状態をコミットしない。
- コミットメッセージは「何を・なぜ」が分かるように書く。
- `.env`・秘密情報・巨大な生成物をコミットしない。
- **履歴改変（force push / rebase / reset --hard）はユーザー承認なしで行わない。**
- ブランチ運用がある場合、main / master への直接コミットは避け、作業ブランチを使う。

---

## 8. コンテキスト・効率の規律

- 迷ったらファイル全読みではなく、検索で該当箇所を絞ってから読む。
- 同じ情報を何度も取得しない。分かった事実は前提として次の判断に使う。
- 長時間の作業では、区切りごとに「完了したこと・残っていること」を短く整理する。
- ユーザーへの報告は結論から書き、検証結果（証拠）を添える。

---

## 9. 停止するべき場面

以下の場合は作業を止めてユーザーに確認する。

- 要求と既存仕様が矛盾する。
- 想定より影響範囲が大きい（DB構造・API仕様・ディレクトリ構成に波及する）。
- `AGENTS.md` §16 の事前確認対象に該当する操作が必要になった。
- 同一問題で3回失敗した（§4）。
- ループ開発が3周で収束しなかった（§6）。
- 仕様の解釈が複数あり、どれを選ぶかで成果物が変わる。
